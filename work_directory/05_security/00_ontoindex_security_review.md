# Security Review: ontoindex (ontograph/ontoindex @ b373fdf)

**Дата:** 2026-06-12
**Метод:** полный аудит сторонней кодовой базы перед локальным вендорингом; 6 параллельных
независимых аудиторов (read-only) по направлениям: supply chain, сеть/эксфильтрация,
выполнение команд, поверхность MCP-инструментов, HTTP API/веб-UI, инструкции для агентов.
**Объект:** монорепо `ontoindex/` (submodule), 3226 файлов (1484 TS; .py/.java/.rs и пр. —
фикстуры мультиязычных парсеров).
**Threat model:** локальный запуск на Windows-машине разработчика; индексация ПРИВАТНЫХ
кодовых баз; MCP-сервер подключён к Claude Code; план — забрать код локально.

---

## ВЕРДИКТ

**Кодовая база пригодна для локального вендоринга и использования.** Бэкдоров,
обфусцированного кода, скрытой телеметрии в исходниках, авто-апдейтов и инъекций
в инструкциях для агентов НЕ обнаружено. Критических и high-уязвимостей в коде нет.
Дефолтная конфигурация — local-only (127.0.0.1, stdio MCP) и read-only.

**Главный ответ на главный вопрос:** индексация вредоносного репозитория НЕ ведёт к
выполнению кода и к записи вне `.ontoindex/` — ontoindex не исполняет ничего из
индексируемого репо (нет хуков, нет конфигов с командами, грамматики парсеров захардкожены),
все вызовы git/rg идут через `execFile` с массивами аргументов, `shell:true` не используется
нигде, `eval`/`new Function`/`vm` отсутствуют.

Приватный код МОЖЕТ покинуть машину ТОЛЬКО при явном opt-in пользователя (LLM-фичи) —
см. находку F2.

## НАХОДКИ (по убыванию важности)

### F1. MEDIUM — телеметрия `@scarf/scarf` в зависимостях
- `ontoindex/package.json` объявляет `@scarf/scarf@^1.4.0` (`hasInstallScript: true`) —
  пакет аналитики Scarf, шлёт метрики при `npm install`.
- В исходном коде НЕ импортируется нигде (мёртвая зависимость).
- **Мера (обязательная при вендоринге): удалить из package.json до `npm install`**
  (или ставить с `--ignore-scripts`, но это сломает сборку нативных парсеров — лучше удалить пакет).

### F2. MEDIUM — opt-in LLM-фичи отправляют код наружу
- `ontoindex wiki` (генерация вики): шлёт сниппеты кода и контекст в OpenRouter (дефолт) /
  OpenAI / Azure / Z.AI — `src/core/wiki/llm-client.ts`. Требует явный API-ключ.
- Embeddings: при `ONTOINDEX_EMBEDDING_URL`+`ONTOINDEX_EMBEDDING_MODEL` код шлётся на
  указанный endpoint (`src/core/embeddings/http-client.ts`). По умолчанию выключено.
- Web-UI чат: при настройке ключа в UI.
- API-ключи хранятся ОТКРЫТЫМ ТЕКСТОМ в `~/.ontoindex/config.json`.
- **Мера: не настраивать API-ключи; не использовать `wiki` с LLM-провайдерами;
  не задавать ONTOINDEX_EMBEDDING_URL. Тогда CLI/MCP не делает НИ ОДНОГО исходящего вызова.**

### F3. MEDIUM — CDN-ресурсы в генерируемом wiki-HTML и веб-UI
- `src/core/wiki/html-viewer.ts:102-108`: сгенерированный `index.html` грузит
  marked/mermaid/dompurify с cdn.jsdelivr.net.
- `ontoindex-web/index.html:7-11`: Google Fonts.
- Код не утекает, но браузер ходит на внешние CDN (IP-лог, supply chain CDN-скриптов).
- **Мера: при использовании wiki-HTML/веб-UI офлайн — забандлить библиотеки локально (патч).
  Для нашего сценария (MCP/CLI) не критично.**

### F4. MEDIUM — git argument injection (design weakness, не RCE)
- `src/mcp/local/backend-detect-changes.ts:151`: MCP-параметр `base_ref` уходит в
  `git diff <base_ref>` без `--end-of-options` и валидации (контраст: `verification-gap.ts:177`
  делает правильно). Git отвергает мусорные опции с ошибкой — RCE нет, но гигиену стоит поправить.
- **Мера (наш локальный патч): `diffArgs = ['diff', '--end-of-options', params.base_ref, '-U0']`.**

### F5. HIGH (присуще классу инструментов) — prompt-injection через вывод инструментов
- MCP-инструменты возвращают код/комментарии индексируемого репо в контекст Claude БЕЗ
  фильтрации. Вредоносный комментарий в индексируемом коде = инъекция в контекст агента.
  Это неустранимое свойство любого инструмента чтения кода (как и Read/Grep самого Claude Code).
- **Мера: индексировать только доверенные репозитории; не включать `--confirm-writes`
  на чужом коде.**

### F6. LOW/INFO — HTTP API (`ontoindex serve`)
- По умолчанию 127.0.0.1:4747, Bearer-токен (constant-time сравнение), CORS-allowlist
  (localhost:* + ontoindex.vercel.app), SSRF-защита в git-clone (блок приватных IP и
  metadata-endpoints), Cypher write-запросы блокированы, XSS в веб-UI закрыт DOMPurify
  (foreignObject исключён).
- Нет: TLS, rate-limit, ротации/истечения токена, audit-лога. Для localhost-only — приемлемо.
- В Docker веб-UI биндится на 0.0.0.0:4173 (docker-server.mjs:6).
- **Мера: не выставлять порт наружу; Docker-вариант нам не нужен.**

### F7. INFO — MCP-сервер: защита по умолчанию хорошая
- Транспорт ТОЛЬКО stdio (нет сетевого слушателя) — `src/mcp/server.ts`.
- Записи (rename/replace/extract/move) — двойной гейт: флаг CLI `--confirm-writes`
  (по умолчанию ВЫКЛ) + параметр `confirm:true` в вызове.
- Path traversal закрыт: `path-util.ts` canonicalize() — containment к корню репо.
- Cypher: read-only (CREATE/DELETE/SET/... блокированы), обязательный LIMIT ≤ 5000,
  у LadybugDB нет функций файлового IO.

### F8. INFO — supply chain чистый
- 5 package-lock: 99.8% зависимостей с registry.npmjs.org; исключения:
  `tree-sitter-dart` — git-зависимость, ЗАПИНЕНА на коммит 80e23c0 (optional);
  `tree-sitter-proto` — вендорено локально (`file:`).
- Нативные модули собираются из исходников (node-gyp); prebuilt-бинарей в репо нет;
  ontoindex-native — Rust/NAPI, только исходники.
- postinstall-скрипты прочитаны построчно: патч binding.gyp tree-sitter-swift и сборка
  tree-sitter-proto — легитимные, без загрузок и обфускации.
- Обфускации (eval/atob/fromCharCode/base64-блобы) — 0 находок. Авто-апдейтов и
  version-check — 0. Husky pre-commit — только lint/typecheck.
- `@ladybugdb/core` (граф-БД) — обычная npm-зависимость с registry.

### F9. INFO — инструкции для агентов (plugin/skills/hooks) чистые
- Прочитаны ВСЕ агентские файлы: skills (7 SKILL.md), plugin.json, marketplace.json,
  .mcp.json, hooks (ontoindex-hook.js, augment-shell.sh), AGENTS.md, CLAUDE.md, .cursor/*.
- Нет инструкций: фетчить и исполнять удалённый контент, обходить подтверждения,
  игнорировать safety, слать данные наружу. Скрытого юникода (zero-width, bidi),
  скрытых HTML-комментариев — нет. Хуки только ДОБАВЛЯЮТ контекст (augment), ничего
  не исполняют автоматически; паттерн в hook передаётся через `spawnSync` массивом + `--`.
- Нюанс: plugin/cursor-хуки зовут `npx -y ontoindex@latest` — при вендоринге заменить
  на наш локальный путь (иначе зависимость от npm-реестра и «latest»).

## ЧЕК-ЛИСТ БЕЗОПАСНОГО ВЕНДОРИНГА (для нашего скила)

1. Удалить `@scarf/scarf` из `ontoindex/package.json` ДО установки зависимостей.
2. Не настраивать LLM API-ключи; не использовать `wiki`-генерацию с внешними провайдерами;
   не задавать `ONTOINDEX_EMBEDDING_URL`.
3. MCP запускать stdio, БЕЗ `--confirm-writes` (read-only) — наш сценарий «минимизация
   ошибок Claude» и не требует записи через MCP.
4. В хуках/конфигах заменить `npx -y ontoindex@latest` на локальный путь к вендоренной копии.
5. Локальный патч F4 (`--end-of-options` в detect-changes) — желательно.
6. HTTP API (`serve`) и веб-UI не запускать без необходимости; если запускать — только 127.0.0.1.
7. Индексировать только доверенные репозитории (F5).
8. Зафиксировать версию: вендорим срез @ b373fdf, обновления — только осознанным апдейтом
   с повторным diff-ревью.
