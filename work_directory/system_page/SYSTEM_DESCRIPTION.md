# Кодовая память (memory_code_active) — описание системы

> Источник правды для страницы `index.html`. Скоуп — САМА система (что делает, как устроена,
> как работает и эксплуатируется), НЕ история её разработки. Без план-факта, задач и прогресса.

## Простыми словами

**Кодовая память** — это «карта» вашей кодовой базы, которую Claude держит перед глазами,
когда правит код. Вместо того чтобы искать функцию текстовым поиском и править её «вслепую»,
Claude спрашивает у карты: где определён символ, кто его вызывает, что сломается при изменении.

Технически это **локальный граф кода**: парсер Tree-sitter разбирает исходники на символы
(функции, классы, методы) и связи между ними (кто кого вызывает, кто что импортирует), кладёт
их в граф-базу LadybugDB, и отдаёт этот граф Claude через набор инструментов (MCP). Всё —
**локально на машине**, без обращения к сети.

Это **третий слой памяти** в линейке: краткосрочная (`memory_short_active`, контекст диалога) →
долгосрочная (`memory_long_active`, курируемое знание-вики) → **кодовая (структура кода)**.
Первые два слоя помнят *разговор и знание*; кодовая память помнит *структуру кода*.

## Архитектура: движок + проект

Система разделена на две части:

- **Движок — один на машину**, в `~/.claude/tools/ontoindex/`. Это вендоренный (живущий
  локально, независимо от исходного репозитория) срез индексатора ontoindex с парсерами
  Tree-sitter под десяток языков. Ставится один раз скриптом `install.ps1`; нативные модули
  собираются один раз.
- **Артефакты в проекте** — лёгкие, своя копия в каждом проекте: индекс `.ontoindex/` (граф),
  регистрация MCP-сервера в `.mcp.json`, хуки в `.claude/settings.json`, инструкции в `CLAUDE.md`.

Один тяжёлый движок переиспользуется всеми проектами; в проекте лежит только индекс и настройки.

## Конвейер индексации (как строится граф)

```
исходный код  →  Tree-sitter (парсеры)  →  символы + связи  →  граф LadybugDB (.ontoindex/)  →  MCP-инструменты  →  Claude
```

1. `analyze` обходит файлы проекта (с учётом `.ontoindexignore` — что пропускать).
2. Tree-sitter разбирает каждый файл в дерево, из него извлекаются **узлы** (функции, классы,
   методы, файлы) и **рёбра** (CALLS, IMPORTS, и пр.).
3. Граф сохраняется в LadybugDB внутри `.ontoindex/` + метаданные (`meta.json`, последний коммит).
4. MCP-сервер отдаёт граф Claude по запросу.

Пример масштаба: средний проект ~3000 файлов → десятки тысяч узлов и рёбер за 1–2 минуты.

## MCP-инструменты (что Claude спрашивает у графа)

Сервер работает по транспорту **stdio** (никакого сетевого порта) и **только на чтение**
(записи в код через MCP отключены — правки Claude делает своими обычными инструментами).
Доступно ~60 инструментов, ключевые фасады:

- **search** — найти символы (по графу/паттерну), карта репозитория;
- **inspect** (context) — полный контекст символа: кто вызывает, кого вызывает, в каких потоках;
- **impact** — радиус изменения: что сломается при правке функции, уровень риска (LOW…CRITICAL);
- **audit** — архитектурные проверки (мёртвый код, пробелы в тестах и пр.);
- плюс узкие `gn_*` (explore, explain_module, safe_edit_check, pre_commit_audit и др.).

## Рабочий цикл правки (зачем это нужно)

Правило для Claude (ставится в `CLAUDE.md` проекта при активации):

- **Найти** определение/использования → `search` / `inspect`, а не слепой Grep.
- **Перед правкой** функции/класса → `impact`: посмотреть, кто сломается.
- **Перед рефакторингом** → `inspect` + callers/callees.
- **Перед коммитом** → проверить, что изменения затронули только ожидаемое.

Цель — **меньше ошибок «правки вслепую»**: Claude видит последствия до изменения, а не после.

## Хуки (живые подсказки)

Два хука встраиваются в сессию Claude (и сосуществуют с хуками других памятей, не затирая их):

- **augment** (PreToolUse, на Grep/Glob/Bash): когда Claude всё же ищет текстом, хук тихо
  подмешивает к результату граф-контекст по тому же символу — мягкое направление к графу.
- **stale-детект** (PostToolUse, на git-команды **и правки** Edit/Write/MultiEdit): после
  commit/merge сравнивает `git HEAD` с отметкой в индексе; а после серии правок (с порогом и
  кулдауном, чтобы не шуметь) напоминает, что граф их ещё не отражает. И там, и там вывод —
  лишь подсказка переиндексировать (`update`), без авто-reindex: внутри сессии он невозможен
  (MCP держит write-lock БД, а свежий граф видит только новая сессия).

Плюс **инструкционный lazy-reindex**: блок в `CLAUDE.md` велит Claude перед тем как доверять
графу проверить свежесть инструментом `gn_ensure_fresh` (read-only, сверяет индекс с HEAD) и,
если устарел, предложить `update`. Для файлов, правленных в текущей сессии, — доверять файлу,
не графу.

## Режимы (управление через скил `/memory_code_active`)

Все файловые операции выполняет детерминированный скрипт `memory_code.ps1` (не «промпт-правки»):

| Режим | Что делает |
|---|---|
| `on` | строит индекс, регистрирует MCP, ставит хуки + инструкции, правит .gitignore, кладёт .ontoindexignore |
| `off` | снимает MCP/хуки/инструкции; индекс сохраняется (повторный `on` дёшев) |
| `update` | переиндексация до текущего состояния кода |
| `reapply` | обновляет регистрацию (хуки/инструкции/.mcp.json) без переиндексации — донести апдейт скила в активный проект (нет reindex → нет DB-lock) |
| `clear` | убирает индекс в `.trash` (восстановимо) |
| `clear --hard` | удаляет индекс окончательно (с подтверждением) + чистит запись в глобальном реестре |
| `status` | показывает, что активно (индекс/MCP/хуки/инструкции) |

Важно: MCP-инструменты и инструкции из `CLAUDE.md` подхватываются **при старте сессии** —
после `on` нужна **новая сессия** Claude.

## Где что лежит (хранилища)

```
~/.claude/tools/ontoindex/        движок (один на машину): парсеры, dist, скил-скрипты, VERSION.txt
~/.claude/skills/memory_code_active/   скил Claude Code (диспетчер режимов)
~/.ontoindex/registry.json        глобальный реестр проиндексированных проектов (JSON-массив)
<проект>/.ontoindex/              граф проекта (в .gitignore)
<проект>/.mcp.json                регистрация MCP-сервера ontoindex (read-only)
<проект>/.claude/settings.json    хуки augment + stale-детект (вперемешку с чужими — не конфликтуют)
<проект>/CLAUDE.md                блок инструкций MEMORY_CODE
<проект>/.ontoindexignore         что не индексировать (node_modules, dist, build…)
```

## Барьеры безопасности

Система спроектирована «локально-первой» и закрыта несколькими барьерами (env-гейты ставит
режим `on` в `.mcp.json`):

- **MCP только на чтение** — без `--confirm-writes`; код через MCP не меняется.
- **Семантика/эмбеддинги отключены** — `ONTOINDEX_DISABLE_SEMANTIC=1`: иначе движок при первом
  семантическом запросе скачал бы модель с huggingface; гейт бросает ошибку до сети
  (+`HF_HUB_OFFLINE`, `TRANSFORMERS_OFFLINE`).
- **Телеметрия и лог запросов выключены** — `ONTOINDEX_TOOL_TELEMETRY=0`, `ONTOINDEX_QUERY_LOG=0`.
- **Только доверенные репозитории** — индекс читает весь код, и вывод инструментов попадает в
  контекст Claude; вывод трактуется как **данные, не инструкции** (защита от prompt-injection).
- **Никаких внешних запросов** при `analyze`/работе MCP (проверено: ноль исходящих соединений).

## Интеграция с управлением проектами

При создании нового проекта через `/workflow_global_plan` фаза init задаёт вопрос
«Активировать кодовую память?». При «да» система разворачивается автоматически (для пустого
проекта активация откладывается до появления кода).

## Инвентарь диаграмм страницы (выведен из системы)

1. **00 / Линейка** — три слоя памяти (short / long / **code**), место кодовой памяти.
2. **01 / Конвейер** — код → Tree-sitter → граф LadybugDB → MCP → Claude (живой поток).
3. **02 / Движок и проект** — один движок на машину ↔ лёгкие артефакты в проекте.
4. **03 / Цикл правки** — Claude → impact/context → безопасная правка (поток «перед изменением»).
5. **04 / Хуки** — augment (Grep→граф) и stale-детект (commit/правки→напоминание; + lazy-reindex через `gn_ensure_fresh`).
6. **05 / Режимы** — таблица on/off/update/reapply/clear/clear --hard/status.
7. **06 / Хранилища** — дерево, где что лежит.
8. **07 / Барьеры** — гейты безопасности.
9. **Легенда** — как читать схемы.
