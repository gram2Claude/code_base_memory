# Спека проекта: code_base_memory

**Статус: СОГЛАСОВАНО ПО Q1–Q5 (ответы управленца 2026-06-12, см. внизу); Q6 — выбрать на чекпоинте плана.**
**Дата:** 2026-06-12

## 1. Цель

Универсальный, переносимый «скил памяти кодовой базы» для Claude Code: в любом проекте, где
это необходимо, одной командой активируется локальный графовый индекс кода (ontoindex:
Tree-sitter → LadybugDB → MCP), и Claude при работе с кодом обязан использовать его
возможности (поиск символов, контекст, impact-анализ, граф зависимостей), чтобы
минимизировать ошибки при правках рабочих файлов, функций, классов.

По аналогии с уже существующими скилами пользователя:
`memory_short_active` (краткосрочная) и `memory_long_active` (долгосрочная Wiki-LLM) —
это ТРЕТИЙ слой: **память о структуре кода** (рабочее имя скила: `memory_code_active`, Q4).

## 2. Результаты / Deliverables

1. **Вендоренный срез ontoindex** — живёт в нашем репо/инфраструктуре, НЕ зависит от
   github.com/ontograph/ontoindex (способ — Q2). С применёнными мерами безопасности из
   ревью (`05_security/00_ontoindex_security_review.md`): удалён @scarf/scarf, патч
   `--end-of-options` (F4), хуки переведены с `npx ontoindex@latest` на локальный путь.
2. **Скил `memory_code_active`** (Q4) с режимами как у братьев-скилов:
   - `on` — развернуть в текущем проекте: установить/слинковать движок (Q3), построить индекс
     (`analyze`), зарегистрировать MCP-сервер (stdio, БЕЗ `--confirm-writes`), установить
     инструкции для Claude (правила использования ontoindex при работе с кодом);
   - `off` — снять MCP/инструкции, индекс сохранить;
   - `clear` / `clear --hard` — очистить индекс (.ontoindex/) мягко/окончательно;
   - `update` — переиндексация после изменений кода (и/или хук-напоминание о stale-индексе).
3. **Интеграция с `/workflow_global_plan`**: при вызове команды (фаза init нового проекта)
   появляется вопрос «Активировать кодовую память для этого проекта?» — при «да» скил
   разворачивается автоматически; решение фиксируется в config.json проекта.
4. **Правила использования для Claude** (инструкции/skill): при работе с кодом в проекте
   с активной кодовой памятью Claude обязан перед правками функций/классов использовать
   ontoindex-инструменты (impact, context, search, callers/callees) вместо слепого Grep/Read;
   после merge/commit — обновлять индекс.
5. **Документация**: README скила (установка, режимы, ограничения), запись в орг-память.

## 3. Критерии приёмки (черновик — подтвердить)

- На чистом тестовом проекте `memory_code_active on` разворачивает всё без ручных шагов;
  Claude в новой сессии видит MCP-инструменты ontoindex и реально использует их при задаче
  «изменить функцию X» (проверяемо по транскрипту).
- Полностью офлайн: после установки НИ ОДНОГО исходящего сетевого вызова (проверка
  перехватом трафика при первом запуске — контрольная задача из security-ревью).
- Удаление апстрим-репо ontograph/ontoindex НЕ ломает ни установку, ни работу скила.
- `/workflow_global_plan init` нового проекта задаёт вопрос про кодовую память и при «да»
  активирует её.
- `off`/`clear` отрабатывают идемпотентно, не ломая остальные памяти проекта.

## 4. Ограничения

- **Лицензия:** ontoindex — AGPL-3.0-or-later. Вендорим с сохранением LICENSE/NOTICE;
  наш репо публичный — модифицированные исходники и так публикуются (совместимо). Скил-обвязка —
  наш код, но в связке с AGPL-кодом распространяется на тех же условиях.
- **Безопасность:** обязателен чек-лист из `05_security/00_ontoindex_security_review.md`
  (8 пунктов): scarf, LLM-ключи, stdio read-only, локальные пути в хуках, F4-патч,
  не выставлять HTTP API, только доверенные репо, фиксация среза @ b373fdf.
- **Платформа:** Windows (PowerShell), Node 22 (nvmrc), нативная сборка tree-sitter
  (node-gyp → нужны build tools); это самый рискованный пункт установки.
- Реестр задач — timechecker (CBM), репо gram2Claude/code_base_memory, ветки oleg → main.

## 6. Риски

- Нативная сборка tree-sitter на Windows (node-gyp, MSVC build tools) — главный риск
  установки «в один клик»; смягчение: один раз собрать глобально (Q3-вариант A) и
  переиспользовать, либо зафиксировать pre-built node_modules.
- Размер вендоренного среза (монорепо ~3200 файлов) — раздувает наш репо; управленец принял
  риск: вендорим ВСЁ целиком (Q5). node_modules и .git в коммит не входят.
- Дрейф апстрима: мы фиксируемся на b373fdf; апдейты — только осознанно с diff-ревью.
- AGPL-осторожность при будущем закрытии репо (если сделать private — для собственного
  использования всё равно ок, AGPL обязательства возникают при распространении/сетевом сервисе).
