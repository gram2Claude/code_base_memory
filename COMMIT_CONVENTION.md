# Конвенция работы с репозиторием code_base_memory

## Ветки

- **`main`** — защищённая целевая ветка. Прямой push запрещён (branch protection).
  Мержит только координатор (ИИ) через merge-гейт после зелёной проверки.
- **`oleg`** — ветка разработчика. Вся работа и все push — только сюда.

## Процесс разработчика

1. Берёшь задачу → пишешь в чат координатору: **«беру `CBM-N` в работу»**
   (N — номер задачи в реестре timechecker). Координатор ставит In Progress.
2. Работаешь в ветке `oleg`, коммитишь и пушишь как удобно — per-commit конвенция
   НЕ требуется (статусы выводятся не из коммитов). Желательно (для гигиены)
   упоминать `CBM-N` в сообщениях коммитов.
3. Готово → пишешь: **«готово к merge, `CBM-N`»** (можно несколько ID —
   спринт обычно поставляется одним merge).
4. Координатор: пробный merge → проверочная лестница (install, build, смоук) →
   согласование с управленцем → merge `--no-ff` в `main` → задачи в Done.

## Особенности проекта

- Задачи типов qa / doc / research закрываются БЕЗ merge — по верификации/готовности
  артефакта (отчёт в `work_directory/04_reviews/` или `05_security/`).
- Вендоренный срез `ontoindex/` (AGPL-3.0, апстрим-коммит b373fdf) правится ТОЛЬКО
  в рамках задач плана (security-патчи t02/t19); LICENSE и NOTICE не удалять.

## Проверка перед «готово к merge»

- сборка движка зелёная (`npm install && npm run build` в пакете ontoindex/), если задача её касается;
- критерии готовности задачи (done_criteria в каноне) выполнены.

## Артефакты планирования

`work_directory/00_global_plan/` — глобальный план (канон, витрина, ревью) — ведёт координатор,
руками не править. Спеки — `work_directory/01_specs/`, планы реализации — `02_plans/`,
отчёты QA — `04_reviews/`, безопасность — `05_security/`.
